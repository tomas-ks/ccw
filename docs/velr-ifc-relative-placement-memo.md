## Summary

Before the upstream fix, `IfcLocalPlacement.RelativePlacement` was not
materialized into the imported Velr graph for the `building-architecture`
fixture.

This is separate from the earlier GraphQL storage-contract mismatch. In the
current aligned stack:

- the runtime GraphQL schema says `IfcLocalPlacement.RelativePlacement` is a
  stored `RELATIVE_PLACEMENT` relationship
- the generated import artifacts do not emit any `RELATIVE_PLACEMENT` edges for
  `IfcLocalPlacement`
- the live imported database therefore contains placement-parent chains but no
  local placement frames

At the time, `w` had placement resolution code ready, but it had to fall back
to identity transforms because the imported graph was missing the required
relationship.

## Why This Does Not Look Like `w` Misuse

This historical repro was strong because all three artifacts came from the same
import flow:

1. the runtime schema generated by the import says the field is a stored edge
2. the generated import bundle for that same model omits the edge
3. the live database imported from that bundle therefore also omits the edge

So this did not look like `w` querying the wrong relationship name or
misreading the runtime contract.

## Impact On `w`

`w` had local placement chain resolution code in `cc-w-velr`, but it could not
compute object-to-world transforms from the imported graph yet because
`IfcLocalPlacement` nodes do not point at their relative coordinate frames.

Behavior on our side at that time was therefore:

- use the expected `RELATIVE_PLACEMENT` contract
- resolve parent placement chains when possible
- fall back to identity transform when the relationship is missing

That kept rendering alive, but object placement was wrong for real IFC content
until the importer emits the missing edge.

## Suggested Upstream Fix

The upstream fix needed to ensure that explicit field
`IfcLocalPlacement.RelativePlacement : IfcAxis2Placement` is emitted into the
graph as:

```text
(lp:IfcLocalPlacement)-[:RELATIVE_PLACEMENT]->(axis:IfcAxis2Placement*)
```

where `axis` may be `IfcAxis2Placement2D` or `IfcAxis2Placement3D` depending on
the source data.

## Suggested Upstream Acceptance Test

The upstream acceptance test we wanted was:

1. import the IFC into a fresh Velr database
2. assert that `IfcLocalPlacement` count is non-zero
3. assert that at least one `RELATIVE_PLACEMENT` edge exists
4. assert that a query walking `IfcLocalPlacement -> RelativePlacement ->
   Location` returns rows

That should catch this regression directly at the import/runtime boundary.
